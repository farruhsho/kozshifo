## 0.1 Ниша

Частная офтальмологическая клиника (KO'Z SHIFO, Фергана, Узбекистан).
Оплата наличными/картой/QR на месте, без страховых. Один основной филиал,
мульти-филиальность заложена. Приезжающие хирурги из Ташкента. Поток —
живая очередь (не запись): пациент приходит → платит → сидит перед ТВ-табло.

**Что заменяет система:** бумажный журнал регистратуры, бумажные карты 025-8,
тетрадь кассы, Excel склада, выкрикивание фамилий в коридоре.

**Успех для клиники:** пациент проходит регистрация→оплата→диагностика→врач→
лечение/операция без повторного ввода данных и без «потерявшихся» людей;
владелец в реальном времени видит деньги, склад, загрузку и должников;
ни одно назначение врача не теряется.

## 0.3 Журналы актёров (цели · действия · состояния · краевые случаи)

### 2. Администратор фронт-офиса
**Цели:** быстро зарегистрировать/найти пациента, принять деньги, выдать номер,
не создать дубль, закрыть смену без недостачи.
**Действия:** регистрация (телефон-first, антидубликат ✅), корзина услуг,
скидка (% XOR сумма + причина ✅), оплата (нал/карта/QR, split ✅), возврат
(guarded ✅), направление (диагностика|врач|без ✅), печать чека+номера
(автопечать ✅), отмена визита ✅, долги: список должников + погашение вкл.
частичное ✅, закрытие смены + CSV ✅.
**Краевые случаи:**
- Дубль пациента при регистрации — диалог «Возможные дубликаты» ✅ (C4)
- Двойной клик по «Оплатить» → двойная оплата? `?` (H4a идемпотентность)
- Частичная оплата: выдаётся ли номер очереди / когда? `?` (H4 — правило)
- Возврат после выполнения операции — деньги назад, расходники нет ✅ (E1/C9c)
- Возврат «отвязывает» оплаченные строки, чтобы операцию можно было отцепить ✅ (C9a)
- Пациент ушёл не дождавшись — отмена визита ✅ + авто-отмена операций ✅
- Пересменка двух кассиров в один день — смена делится по кассирам? `?` (H14)
- 100 % скидка — путь без денег, но с номером ✅

### 3. Диагност
**Цели:** вызвать следующего, снять показания, прикрепить результат, передать врачу.
**Действия:** call-next по своей дорожке D- ✅, повторный вызов/пропустить/оставить ✅,
внести заключение из справочника разрешённых ✅, прикрепить файл (УЗИ PDF) ✅,
привязать «сиротский» результат прибора ✅, done → авто-V-талон к врачу ✅.
**Краевые случаи:**
- 2+ диагностические услуги в визите — open pool, оба спеца видят ✅ (C2)
- Результат прибора пришёл без пациента/визита — unlinked + link ✅
- Привязали результат НЕ к тому пациенту — unlink/relink есть? `?` (H10 — медицинская ошибка!)
- Талон done, а вложения нет — ловится «зависшими» cat3 ✅
- Пациент не откликнулся на вызов — «Оставить» → назад в waiting ✅

### 4. Врач-офтальмолог
**Цели:** видеть пациента целиком (3 колонки), решить: лечение / операция /
повторный приём / завершить.
**Действия:** call-next V- (свои маршрутизированные первыми ✅), осмотр 025-8
(автосейв ✅, подтяжка рефракции с RMK-700 ✅), диагнозы (мульти ✅, чипы ✅),
назначить лечение (билл через услугу ✅ C5), направить на операцию (хирург вкл.
внешнего ✅), «Завершить приём» от визита ✅ (Ф4-фикс), печать карты 025-8 ✅.
**Краевые случаи:**
- Пациент пришёл сразу к врачу (без диагностики) — referral_intent=doctor ✅
- «Завершить приём» без активного талона — не блокирует ✅
- Повторный приём: follow_up проставлен — а дальше? Кто и как возвращает
  пациента? Recall-лист «кому пора» есть? `?` (H1 — потерянные follow-up)
- Рецепт на очки / медикаменты — печатная форма рецепта есть? `?` (H3)
- Две consult-услуги к РАЗНЫМ врачам в одном визите — маршрутизация? `?` (H9)

### 5. Хирург
**Цели:** видеть свой операционный день, выполнить, зафиксировать расходники, закрыть счёт.
**Действия:** доска дня (referred-pool → place/detach ✅), календарь-агенда ✅,

**Краевые случаи:**

## 0.4 Выведенная модель (сверена с кодом на верхнем уровне)

**Сущности:** Patient (регион/район, лечащий врач, антидубликат) · Visit
(flow_status — событийная машина, скидка, payable) · VisitItem (строки счёта,
status ordered/paid) · Payment/Refund · QueueTicket (D-/V-/С-/Л-, assigned_user,
room) · EyeExam (025-8) · VisitDiagnosis (каталог) · Attachment (uzi/hiv/lab/other)
· DeviceResult · Operation (referred→scheduled→in_progress→performed→completed/
cancelled + financially_closed_at) · Treatment (+service_id биллинг) · Product/
Batch/StockMovement (FEFO) · Expense/ExpenseType · Payroll · Debt (представление,
без таблицы) · User/Role/Permission (динамические) · Cabinet · Branch · AuditLog ·
UserSession · Notification (журнал доставки) + insights computed-on-read.

**Машина визита (события, не ручные статусы):** registered → waiting_diagnostic →
in_diagnostic → awaiting_assignment|waiting_doctor → in_doctor → (треки
Лечение/Операция) → follow_up | completed | cancelled.

**Ключевые правила:** оплата двигает flow и рождает номер; done диагностики
рождает V-номер (маршрутизация по услуге, балансировка); операция биллится при
направлении, цена гибкая до финзакрытия; расходники списываются при perform
(FEFO, атомарно); возврат откатывает paid→ordered; визит закрывается только без
активных номеров/операций/назначений; все мутации в аудите.

**Почему H9 отложено:** модель `service_doctors` (M2M) задаёт, какие врачи *могут*
оказать услугу, а не «визиту нужно N отдельных консультаций». `_eligible_doctor_
for_visit` намеренно union'ит врачей по всем оплаченным услугам и при нескольких
кандидатах БАЛАНСИРУЕТ нагрузку к одному (это правильно для частого случая: услуга,
доступная нескольким врачам, → к наименее загруженному). Настоящий фан-аут «пациент
идёт к врачу A И к врачу B» требует нового понятия «консультация на услугу» + флоу
с параллельными врачебными ногами (appointment_finished должен срабатывать лишь
когда закрыт ПОСЛЕДНИЙ активный V-талон) — это редизайн, а не быстрый фикс. Ручной
обход существует: после закрытия первого приёма регистратура «Направляет к врачу»
второго. Отложено как осознанный редизайн, а не тупик.
